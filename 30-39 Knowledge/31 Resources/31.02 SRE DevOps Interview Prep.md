# 第 1 轮：基础深度 + 故障排查

## 题 1：Linux 内核 — `%sy` 高排查

### 1.1 `%sy` 高通常是什么原因？（≥ 3 个方向）

| 方向 | 具体原因 | 机制 |
|------|---------|------|
| **频繁系统调用** | 代码里大量 `read/write/send/recv`，尤其是小块 IO | 用户态切内核态是有开销的，每次切换几十到几百 CPU cycle，累积起来很可观 |
| **上下文切换过多** | 线程数过多（比如 1000+ 线程），频繁争抢锁 | 上下文切换本身走内核调度器，算 sys time；且切换后的 Cache Miss 会让 user time 也被拖慢 |
| **内存映射/缺页** | JVM 堆太大 + 没开 HugePage + `mmap` 频繁 | 缺页中断（page fault）是内核处理的，大量的 minor page fault 会推高 sys |
| **中断处理** | 网卡中断太多（小包流量），中断亲和性没配好 | 软中断（softirq）处理网络收包，全算在 sys |
| **文件系统操作** | 日志写盘、`fsync`、大量小文件创建删除 | 文件系统操作走 VFS + 具体 FS 的内核代码 |

### 1.2 怎么一步步定位到根因？

```bash
# Step 1: 先看清楚 CPU 时间都花在哪了
mpstat -P ALL 1 5
# 每颗核的 %sys %usr %iowait %irq %soft

# Step 2: 用 perf 看内核态热点函数
perf top -g
# 或 perf record -g -p <pid> -- sleep 30 && perf report

# Step 3: 统计系统调用分布
strace -c -p <pid> -f
# 直接看到哪个系统调用耗时最多

# Step 4: 如果是上下文切换导致
vmstat 1 5
pidstat -w -p <pid> 1

# Step 5: 看中断分布
cat /proc/interrupts
```

### 1.3 `write()` 占比大但磁盘 IO 不高，怎么解释？

`write()` 系统调用写的是**内核的 Page Cache**，不是直接写磁盘。

```
应用程序 write()
    ↓
内核 Page Cache（内存，很快）
    ↓
后台 pdflush/flush/kworker 线程 → 异步刷到磁盘
```

`write()` 的 CPU 消耗在：
- 用户态 buffer → 内核态 buffer 的内存拷贝（`copy_from_user`）
- Page Cache 的页分配和查找

```bash
# 验证：看脏页量
cat /proc/meminfo | grep -E "Dirty|Writeback"
```

> 另外如果写的是 socket（慢速客户端），`write()` 可能阻塞在 TCP 发送缓冲区满的情况下。`strace` 里看到 `write()` 耗时长，要去确认 fd 到底是文件还是 socket。

## 题 2：TCP/网络 — 偶发超时排查

### 2.1 TCP 层面有哪些可能导致"偶发 1-3 秒超时"？

| 原因 | 机制 | 特征 |
|------|------|------|
| **TCP 重传超时（RTO）** | 包丢了，等 RTO 指数退避后重传 | 同一 Seq 的包被发多次，间隔越来越大 |
| **Nagle + Delayed ACK 互殴** | 发送端等更多数据，接收端等更多 ACK，双方互等 | ~40ms 延迟（叠加可上百 ms） |
| **零窗口（Zero Window）** | 接收端缓冲区满，通告 win=0 | `tcpdump` 看到 `win 0`，之后沉默 |
| **SYN/Accept 队列满** | 连接被丢弃，客户端等 SYN 超时重传 | `ss -s` 看溢出计数 |
| **Keep-Alive 超时断开** | 中间 NAT/防火墙清理了空闲连接 | 发数据时发现连接已死 |
| **客户端端口耗尽** | 大量短连接用完本地端口 | `netstat -an \| grep TIME_WAIT` 巨大 |

### 2.2 怎么用 `tcpdump` 分析？

```bash
tcpdump -i eth0 -nn -s 0 host <ip> and port <port> -w /tmp/tcp.pcap

# Step 1: 看三次握手阶段
tcpdump -r /tmp/tcp.pcap 'tcp[tcpflags] & (tcp-syn|tcp-ack) != 0'

# Step 2: 看数据传输阶段，找超时/重传
tcpdump -r /tmp/tcp.pcap 'tcp[tcpflags] & tcp-push != 0' | head -50

# Step 3: 看结束阶段
tcpdump -r /tmp/tcp.pcap 'tcp[tcpflags] & (tcp-fin|tcp-rst) != 0'

# Step 4: 看 TCP 选项（关键）
tcpdump -r /tmp/tcp.pcap -v
# MSS、Window Scale、SACK、Timestamps
```

### 2.3 大量 SYN 重传但端口在监听？

问题在 Server 回 SYN-ACK 或 Client 收 SYN-ACK 的路上。

**可能原因：**
1. **SYN 队列/半连接队列满**
2. **conntrack 表满**：`conntrack -S`，经典坑 `nf_conntrack: table full, dropping packet`
3. **网络不对称路由**
4. **Server 多网卡，回包走错接口**
5. **ARP 缓存过期**

**SYN Cookie 知识点：**
- 半连接队列满时触发，防御 SYN Flood
- `net.ipv4.tcp_syncookies = 1`
- 代价：Cookie 里放不下 TCP 选项（Window Scale、SACK 等），连接质量下降
- `cat /proc/net/netstat` 里 `TCPReqQFullDrop` 字段会告诉你有没有溢出

## 题 3：容器/K8s — OOM Kill 排查

### 3.1 `-Xmx=1.5Gi` 为什么实际用了超过 2Gi？

**JVM 进程内存 ≠ 堆内存。** JVM 进程 RSS 包含：

```
JVM 进程总内存 =
  Heap（-Xmx控制）
  + Metaspace（-XX:MaxMetaspaceSize，存类元数据）
  + Thread Stacks（-Xss × 线程数，默认 1MB/线程）
  + Direct Memory（-XX:MaxDirectMemorySize，默认等于 -Xmx）
  + Code Cache（JIT 编译的代码）
  + GC 内部数据结构
  + Native memory（JNI 调用）
  + Compressed Class Space + Symbol tables
  + 共享库、mmap 文件
```

**以典型 Spring Boot 应用为例：**

| 组件 | 估算大小 |
|------|---------|
| Heap（-Xmx1.5Gi） | 1.5 Gi |
| Metaspace | 100-200 Mi |
| 200 个线程 × 1MiB stack | 200 Mi |
| Direct Memory（Netty） | 100-300 Mi |
| Code Cache | 50-150 Mi |
| GC overhead | 50-100 Mi |
| **总计** | **~2.0-2.5 Gi** |

### 3.2 K8s OOMKiller 和 Linux OOMKiller 的关系

两者是**同一个机制**（Linux 内核 OOMKiller），触发条件不同：
- K8s 场景：cgroup 级别内存限制触发 cgroup 内 OOM
- 宿主机场景：整机内存不足触发全局 OOM

```bash
# K8s 层面
kubectl describe pod <pod-name> | grep -A 5 "State"
# → Reason: OOMKilled, Exit Code: 137 (128+9=SIGKILL)

# 宿主机层面
dmesg | grep -i "killed process"
```

### 3.3 不加大 limits，JVM 优化方案

```bash
# 1. 调小堆
-Xmx1g -Xms1g

# 2. 限制 Metaspace
-XX:MaxMetaspaceSize=128m

# 3. 限制线程数
server.tomcat.threads.max=50

# 4. 限制 Direct Memory
-XX:MaxDirectMemorySize=128m

# 5. GC 调优
-XX:+UseG1GC -XX:MaxGCPauseMillis=200

# 6. 容器感知（关键！）
-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0
# 用 limits 的 75% 作为堆上限，比硬编码 -Xmx 更安全
```

## 题 4：故障排查实战 — 凌晨 2 点 Page

### 前 3 分钟

```
00:00 手机响了，看到告警
00:01 打开 Grafana 确认真实性：5xx 率 25%，P99 8s，QPS 持平
00:02 发消息到 Incident Slack Channel：
      "P2 事故：登录服务 5xx 率 25%。IC: 我。
       @dev-oncall 请确认最近 4 小时代码发布。"
00:03 决策：拉人还是自己扛
```

### 排查依赖链顺序（从离自己最近的开始）

```
Step 1: 看自己的服务（30s）
  → 线程池 active thread 是否打满？
  → 打满 → 自己没问题，下游拖慢

Step 2: 看延迟分布（1min）
  → 按依赖分组：
    LDAP: 4500ms ❌  Redis: 2ms ✅  MySQL: 5ms ✅
  → 锁定 LDAP

Step 3: 确认不是自己的锅（30s）
  → LDAP 连接池：连接数正常
  → 超时设置：5s → 刚好对应 P99 延迟

Step 4: 通知 LDAP 团队 + 做自己止损
```

### 回滚 vs 排查决策矩阵

| 情况 | 决策 |
|------|------|
| 最近有发布 + 发布前正常 | **直接回滚** |
| 最近无发布 + 预计 10min 能修 | 继续排查 |
| 原因不明 + 已过 15min | 切流量/降级 |
| 已影响用户 30min | **不管原因，先降级再查** |

### 协作分工

| 角色 | 职责 |
|------|------|
| **IC（总指挥）** | 只做决策，每 30-45min 发状态更新 |
| **Ops Lead** | 带人排查修复，执行 IC 决策 |
| **Comm Lead** | 更新 Status Page，通知客服，挡外部压力 |

### 恢复后动作

1. 盯着指标 15 分钟确认不是"假恢复"
2. 确认无脏数据/死信堆积/限流误伤
3. 当天写事故报告初稿（趁记忆新鲜）
4. 48 小时内安排 Postmortem
5. Action Item 写到 Issue Tracker，每周 Review

## 题 5：设计 CI/CD 平台

### 5.1 整体架构

```
┌──────────────────────────────────────────────────────────┐
│                     CI/CD Platform                        │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  Pipeline DSL（集中仓库维护）         Web Portal / API     │
│  团队只需声明语言/构建命令/端口         部署状态/审批/回滚   │
│                                                           │
│  Pipeline Engine: Argo Workflows                          │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐           │
│  │ Lint │→│ Test │→│ Scan │→│Build │→│Deploy│           │
│  │      │ │      │ │(强制)│ │Image │ │      │           │
│  └──────┘ └──────┘ └──┬───┘ └──────┘ └──┬───┘           │
│                        │                 │                │
│            Security Gate（不可跳过）      │                │
│            漏洞扫描+合规检查+SBOM        │                │
│                        │                 │                │
│            Image Registry (Harbor)       │                │
│            镜像签名(Cosign)              │                │
│                        │                 │                │
│            Deployment Controller: ArgoCD                  │
│            金丝雀(Argo Rollouts) + GitOps                 │
│                                                           │
│  基础服务: Vault(密钥) + 远程缓存 + 测试环境池              │
└──────────────────────────────────────────────────────────┘
```

**技术栈选型理由：**

| 组件 | 选型 | 理由 |
|------|------|------|
| Pipeline Engine | Argo Workflows | 原生 K8s，每个 Step 独立 Pod，声明式 |
| GitOps 部署 | ArgoCD | 目标状态在 Git，天然审计 |
| 渐进式发布 | Argo Rollouts | 金丝雀 + 自动指标分析 + 自动回滚 |
| 镜像仓库 | Harbor | 自带漏洞扫描、镜像签名、RBAC |
| 密钥管理 | Vault | 不把密钥放 CI 变量里 |
| 策略引擎 | OPA/Gatekeeper | 安全策略用代码写，可追溯 |
| 安全扫描 | Trivy + SonarQube | 容器漏洞 + 代码质量 + 依赖库漏洞 |

### 5.2 多租户 + 标准化

每个服务仓库下只有一个 `.ci/service.yaml`：

```yaml
name: login-service
team: auth-team
language: java
framework: spring-boot
build:
  command: ./mvnw package -DskipTests
runtime:
  port: 8080
  health_check: /actuator/health
deploy:
  strategy: canary
  canary_steps: [10, 30, 100]
```

| 层 | 谁控制 | 灵活性 |
|-----|--------|--------|
| Pipeline 流程、安全卡点 | **平台控制** | 零灵活 |
| 构建命令、语言版本 | **团队控制** | 灵活 |
| 部署策略 | **团队控制** | 灵活 |
| 镜像仓库、K8s 集群 | **平台控制** | 零灵活 |

### 5.3 安全卡点（多层防御）

```
第 1 层: Pipeline 内置 — SAST + SCA + Trivy + 密钥检测
第 2 层: K8s 准入控制 — OPA/Gatekeeper（无签名/无SBOM 拒绝部署）
第 3 层: 运行时持续检测 — Falco + Trivy Operator
第 4 层: 审计 + 追责 — 紧急通道需 VP 审批
```

### 5.4 渐进式发布

```yaml
strategy:
  canary:
    steps:
      - setWeight: 10
      - pause: {duration: 5m}
      - analysis:         # 自动指标分析
          templates:
            - templateName: canary-analysis
      - setWeight: 30
      - pause: {duration: 10m}
      - setWeight: 100
```

**蓝绿 vs 金丝雀：**
- 蓝绿 = 瞬时切换 100% → 新版本有 bug 全炸
- 金丝雀 = 逐步放量 → 降低爆炸半径
- 蓝绿适合无状态基础设施，业务服务推荐金丝雀

### 5.5 可观测性

```promql
# 部署成功率
sum(deployments_success_total{team="$team"})
/
sum(deployments_total{team="$team"})

# 哪个阶段失败最多
sum by (stage, failure_reason) (rate(deployments_failed_total[7d]))
```

### 5.6 推动 50 个团队迁移

```
阶段 1: 找 2-3 个"友好团队"试点（找有影响力的，不是痛点最深的）
阶段 2: 让他们爽 — 不需维护 Jenkinsfile、自动回滚、一键创建
阶段 3: 让不迁移的自然痛点暴露
阶段 4: 公布试点数据，Manager 们自己来问
阶段 5: 定 Deadline（老 Jenkins 不再维护，出问题团队自负）
阶段 6: 留逃生门（遗留系统可以申请例外）

话术心法：不讲"CI/CD 标准化"，讲"你不再被部署问题叫醒"
```

## 题 6：设计可观测性平台

### 6.1 三支柱技术选型

```
Metrics:  Prometheus + Thanos/Cortex + Grafana
Logging:  OpenTelemetry Collector → Kafka → Loki + Grafana
Tracing:  OpenTelemetry SDK → Collector → Jaeger/Tempo + Grafana
统一入口: Grafana LGTM Stack
```

**为什么这个组合：**

| 决策 | 选型 | 反对方案 | 理由 |
|------|------|---------|------|
| Metrics | Prometheus | Zabbix/Datadog | 云原生标准、Pull 模型对 K8s 友好 |
| 长期存储 | Thanos | 裸 Prometheus | 降采样 + 对象存储，成本可控 |
| 日志存储 | Loki | Elasticsearch | 轻量、和 Prometheus 同标签模型 |
| 日志缓冲 | Kafka | 直写 | 削峰填谷，防后端被打挂 |
| 链路 | Tempo | Zipkin | OTel 原生，和 Grafana 紧密集成 |

### 6.2 三支柱关联

```
Grafana 告警触发
  → 看 Metrics 面板发现 P99 飙升
  → 一键跳转 Tempo（带时间范围）
  → 看 Top N 慢 Trace → 定位到某个 Span 花了 2.8s
  → 一键跳转 Loki（同 trace_id）
  → 看到日志 "LDAP timeout after 3s"
  → 根因定位完毕，3 分钟
```

**关键：Trace ID 出现在 Log/Metrics/Trace 的所有环节。**

### 6.3 日志规范

不靠自觉，靠 SDK + Agent：

```xml
<!-- 内部 SDK: logging-sre-starter -->
<dependency>
    <groupId>com.company.sre</groupId>
    <artifactId>logging-sre-starter</artifactId>
</dependency>
```

```bash
# JVM 启动参数 — 不改代码
java -javaagent:opentelemetry-javaagent.jar \
     -Dotel.resource.attributes=service.name=login-service \
     -jar login-service.jar
```

输出统一 JSON 格式，包含 timestamp/level/message/trace_id/span_id/service/team/pod。

CI 里自动检查：没引入 SDK → 构建失败；自定义 logback → 禁止。

### 6.4 成本控制（5TB/天 × 30 天）

```
热数据（0-3天）: Loki SSD，全量，15TB
温数据（3-7天）: MinIO/S3，全量+压缩3:1，~13TB
冷数据（7-30天）: 只保留 ERROR/WARN + 采样10% INFO，~1TB
总计：~30TB（比全量省 75%）

额外策略：
- DEBUG 日志 → 生产环境直接丢弃
- 健康检查日志 → Collector 层 drop
- 静态资源 CDN 日志 → 不进主力存储
```

### 6.5 告警治理（500 → 10 条）

| 原因 | 占比 | 治理手段 |
|------|------|---------|
| 误报（阈值不合理） | 40% | 调阈值 + `for: 5m` + 双窗口告警 |
| 重复告警 | 25% | Alertmanager 分组 |
| 低优先级当高优先发 | 15% | Page vs Ticket 分级 |
| 没修→反复告警 | 10% | 告警抑制 + 升级机制 |
| 指标抖动 | 10% | `for: 5m` |

**每周 15 分钟告警 Review 会：**
给每条告警打 A/B/C/D 分，D 级（误报）目标 = 0。

## 题 7：同城双活 + 异地灾备

### 7.1 概念区分

| 维度 | 同城双活 | 异地灾备 |
|------|---------|---------|
| 两个机房都服务流量 | ✅ | ❌（备机房 Standby） |
| 机房间延迟 | < 2ms | 20-50ms |
| 数据同步 | 强同步/半同步 | 异步复制 |
| RPO | ≈ 0 | 几秒~几分钟 |
| RTO | < 30s | 几分钟~几小时 |
| 防什么 | 单机房故障 | 城市级灾难 |
| 成本 | ~1.8x | ~1.3x |

### 7.2 MySQL 跨机房方案

| 方案 | RPO | 坑 |
|------|-----|-----|
| 异步复制 | 几秒~几分钟 | 主库挂时备库可能缺 binlog |
| 半同步复制 | ≈ 0 | 备库网络一断→主库被阻塞→延迟升高 |
| MGR（Paxos 多主） | ≈ 0 | 跨机房 >10ms → 事务提交极慢；脑裂风险 |

**务实推荐：**
- 同城（< 2ms）：MGR 单主 + 半同步
- 异地（> 20ms）：异步复制，接受 RPO ≈ 10 秒
- 不要主-主双写——冲突解决是噩梦

### 7.3 流量调度

```
DNS GSLB（粗粒度，1-5 分钟切换）
API Gateway（细粒度，秒级切换）
BGP Anycast（最快但成本高，需要自有 IP 段）

推荐混合方案：DNS GSLB + Gateway 层双保险
```

**故障切换由人决策（IC），不自动触发——防双主脑裂。**

### 7.4 跨机房 5ms 延迟的放大效应

```
1. 分布式事务（2PC）：单事务 6ms+（单机房 0.5ms × 36 倍）
2. Redis 缓存一致性：复制延迟窗口内读到旧数据 → 超卖
3. 分布式锁（Redlock）：5 个节点 × 5ms = 25ms 起
4. 服务间调用链放大：8 个服务 → 5 倍延迟
5. TCP BDP：10Gbps × 5ms = 6.25MB 窗口才能跑满带宽
```

# 第 3 轮：SRE 文化 + 事故管理 + SLO

## 题 8：SLO 场景分析 — 下单链路

### 8.1 SLI 选择：API 成功率 vs 端到端可用性

**选端到端可用性。** API 返回 200 但前端白屏 → 用户仍然下单失败。

务实组合：
- RUM 埋点（真实用户体验）
- 合成监控（定时机审）
- 服务视角 SLI 同时保留做技术诊断

### 8.2 8 个依赖服务 SLO 分配

| 服务 | SLO | 理由 |
|------|-----|------|
| **支付** | 99.99% | 钱相关，资损风险 |
| **库存** | 99.95% | 库存不准 = 超卖 = 资损 |
| **订单** | 99.95% | 最后一步，数据丢了用户疯 |
| 登录 | 99.9% | 不是每次下单都要登录 |
| 商品/购物车 | 99.9% | 影响体验但不丢钱 |
| 优惠券 | 99.5% | 挂了原价买，不致命 |
| Gateway | 99.99% | 全部流量入口 |

### 8.3 第三方支付 SLO 不达标

```
短期：分开记账 + 找供应商索赔
中期：接第二家支付 / 支付降级（待支付订单）
长期：如实调整下单 SLO，自有服务独立考核
```

### 8.4 回应 VP 要 100%

> "与其追求永远做不到的 100%，我建议做三件事：
> 1. 把日常 SLO 从 99.9% 提到 99.95%
> 2. 做好优雅降级——某个环节挂时用户仍能下单
> 3. 确定 Error Budget 底线——碰线 2 分钟自动启动预案
> 这比喊 100% 口号强一万倍。"

## 题 9：事故管理 + Postmortem

### 9.1 事故管理流程

| 角色 | 谁 | 职责 |
|------|-----|------|
| IC | 最资深的人 | 只做决策，每 30min 发更新 |
| Ops Lead | 最熟悉系统的人 | 带人排查修复 |
| Comm Lead | 擅长沟通的人 | 更新 Status Page、挡外部压力 |
| Scribe | 随便谁 | 实时记录时间线 |

**IC 和 Ops Lead 必须分开——人的注意力是单线程的。**

### 9.2 5-Why 分析

```
问题: Redis Failover 后数据库被打挂

Why 1: 为什么数据库被打挂？
  → Failover 后缓存为空，请求穿透到数据库
Why 2: 为什么穿透能打挂数据库？
  → 没限流保护，连接数打满
Why 3: 为什么连接数会打满？
  → 每个实例连接池 50 × 20 实例 = 1000 = DB max_connections
Why 4: 为什么连接池和 DB 上限这么接近？
  → 各团队自己设，没统一规范
Why 5: 为什么没统一规范？
  → 没有平台级治理，上线没人审核连接池配置

根因: 缓存重建无保护 + 连接池缺乏治理（不是"Redis 挂了"）
```

### 9.3 Action Item 闭环

每个 Action Item 必须有 Owner + DDL + 关联依赖。写到公开 Issue Tracker。每周 SRE 周会 Review。逾期 3 周报告 Manager。每月统计完成率。

### 9.4 说服经理投入稳定性

> "上次 40 分钟中断损失约 80 万。修两个根因需 10 人天。如果在双 11 复现呢？给我一个 Sprint，我修的不是一个 bug，是一类故障。"

## 题 10：团队管理 + SRE 文化

### 10.1 On-call 压力导致想离职

- 先分析原因：排期问题？经验不足？统计偏差？
- 私下 1:1 倾听，不指责
- 调整排班确保公平（发版周双人 On-call）
- 降低频率 + Shadowing 陪跑
- 明确退出通道：可以转项目工作，不影响晋升
- **On-call 承受力下降是系统脆弱的信号，不是个人的失败**

### 10.2 开发不愿 On-call

不要强迫，用证据说服：

> "周六晚上你的 bug 引发故障。你不 On-call → 我读你代码 30min 才理解 → 修 2h。你 On-call → 5min 定位 → 15min 恢复。哪种情况你周末体验更好？"

落地：资深开发先 Shadowing 4 周 → 自愿独立 On-call。运转 3 个月后拿 MTTR 数据对比（45min → 12min）。

### 10.3 隐瞒事故的处理

**Blameless 的核心：私下 1:1，问"你觉得报故障的流程有什么问题？"而不是"你为什么没报？"**

可能是系统惩罚了诚实的人——上次报了被公开批评、Postmortem 太耗时、绩效受影响。**作为 Leader 去修系统，不是修人。**

### 10.4 救火死循环

```
救火 → 没时间自动化 → 更容易故障 → 更多救火

打破方法：
1. 可视化代价（65% 时间救火）
2. 单向门：一个 Sprint 全队不接 On-call，只做自动化
3. 每个 Sprint 固定 20% 时间做改进
4. 暂停新功能：先还债，再借钱
```

# 第 4 轮：综合场景 + 压力题

## 题 11：大促备战（双 11）

### 11.1 容量评估

```
Step 1: 建立容量模型
  每服务: 当前 QPS / 单实例容量 / 现有实例数 → 10x 需多少实例

Step 2: 找依赖链瓶颈
  MySQL/Redis/MQ 的最大承载能力？共享依赖会被所有服务放大

Step 3: 理论 ≠ 实际
  锁竞争、上下文切换、连接池排队 → 3x 实例 ≠ 3x 吞吐量
  必须压测验证
```

### 11.2 压测注意事项

1. **数据隔离**：流量打标 + 影子表/影子 Redis（不触发真实支付/短信）
2. **逐级加压**：2x → 5x → 8x → 10x，每级观察 10min
3. **找拐点而非验证目标**：在多少倍时崩了？
4. **压测要有 Stop 条件**：错误率 >5% 自动停止
5. **半夜压**

### 11.3 四级降级方案

```
🟢 正常: 全部功能
🟡 轻度（负载>60%）: 关推荐算法、搜索建议、DEBUG 日志
🟠 中度（错误率>1%）: 关优惠券校验、积分累计、限流 70%
🔴 重度（下单成功率<99%）: 限流 50%、关非核心服务、允许待支付订单
```

**降级开关管理：** 动态配置中心（Nacos/Apollo），实时生效不需重启。大促前一个月建好，前两周演练，当天一页展示所有开关状态。

### 11.4 多级限流（漏斗模型）

```
L1: CDN/WAF — 按 IP/地域限流
L2: API Gateway — 按 UserID / API path / 服务 限速
L3: 应用层 — Sentinel/Resilience4j 线程池隔离+熔断
L4: 数据库层 — 连接数上限+读写分离
```

### 11.5 大促 vs 平时监控

| 维度 | 平时 | 大促 |
|------|------|------|
| 告警阈值 | 正常 | 适当放宽（CPU >95% 才告） |
| 告警分级 | Page+Ticket | 只保留 Page |
| 核心指标 | SLI/SLO 全套 | 下单成功率 + P99 + GMV + DB负载 + MQ积压 |
| 响应 | 5min | 30s |

**下单成功率和支付成功率的告警不调——这是底线。**

### 11.6 封网

- D-14：大促相关代码冻结
- D-7：全量封网（代码/配置/DB变更/中间件升级/网络变更都不允许）
- 例外：安全漏洞需 VP 审批
- 不封：扩容、降压开关演练

### 11.7 超预期 15 倍的应对

```
T+0min: 下单成功率 98.5%，P99 从 200ms→800ms（危险信号）
T+1min: 判断拐点临近 → 执行降级预案
T+3min: 开启 Gateway 限流 + 关非核心逻辑 + 通知业务方
T+15min: CPU 85%→60%，P99 800ms→350ms → 稳住
```

## 题 12：成本优化 — 砍 30%

### 12.1 从账单反向推导

1. 给所有资源打 Tag（team/service/env/cost-center）
2. 开启 Cost Allocation Tags，按 Tag 分拆账单
3. 找异常：哪个服务成本环比暴涨？哪个环境快赶上 Prod？

### 12.2 聪明降本方案（≥ 6 种）

| 手段 | 预计节省 | 风险 |
|------|---------|------|
| 弹性伸缩（晚上关机） | 15-25% | 低 |
| 资源超卖（调低 Request） | 10-20% | 低 |
| 存储分层（冷热分离） | 20-30% | 低 |
| Spot/抢占式实例 | 5-10% | 中（可能被回收） |
| 合并小服务 | 5-10% | 低 |
| 数据库 right-sizing | 10-15% | 低 |
| 网络优化（同 AZ 通信） | 5-15% | 低 |

### 12.3 回应 CFO："不能影响可用性"

> "完全不影响可用性物理上做不到——成本、可靠性、变更速度是铁三角。但 90% 的砍法不影响可用性（弹性伸缩、right-sizing、存储分层）。最后 10% 我用非关键服务吸收风险，核心服务只跑按需实例。给我 2 个月分两阶段执行，每项标识风险等级，你审批后执行。"
