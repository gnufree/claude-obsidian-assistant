# SRE 核心知识总结：SLO、SLI 与 Error Budget

---

## 一、基础概念

### 1.1 核心定义

| 概念 | 全称 | 含义 | 例子 |
|------|------|------|------|
| **SLI** | Service Level Indicator | 服务等级指标——衡量什么的量化指标 | 登录成功率、P95 延迟 |
| **SLO** | Service Level Objective | 服务等级目标——指标要达到什么值 | 30 天内成功率 ≥ 99.95% |
| **SLA** | Service Level Agreement | 服务等级协议——违约的法律/财务后果 | SLO 不达标 → 赔钱/赔券 |

### 1.2 SLO 的完整公式

```
SLO = SLI + 目标值 + 度量窗口 + 合规阈值
```

**完整示例：**

> "**30 天内**，登录服务的**成功率**（非 5xx / 总请求数）必须**≥ 99.95%**，且**任何 1 小时内**成功率不得低于 **99.5%**。"

| 要素 | 值 | 含义 |
|------|-----|------|
| SLI | 成功率 | 衡量什么 |
| 目标值 | ≥ 99.95% | 多好算好 |
| 度量窗口 | 30 天 | 在哪段时间内算账 |
| 合规阈值 | 单小时不低于 99.5% | 防止"平均值掩盖波动" |

### 1.3 各类型服务的常见 SLI

| 服务类型 | 推荐 SLI |
|---------|---------|
| 同步 API（登录、查询） | 可用性（成功率）、延迟（P95/P99） |
| 异步任务（报表生成） | 新鲜度（数据多久可用）、吞吐量 |
| 存储服务 | 持久性（丢了几个对象）、延迟 |
| 批处理管道 | 覆盖率（处理了多少数据）、时效性（几点前跑完） |

### 1.4 为什么不能承诺 100%

1. **物理定律限制**：网络分区、硬件故障、电力中断——概率永远不会是零
2. **成本呈指数增长**：每多一个 9，成本约是前一个 9 的 10 倍
3. **用户感知不到差异**：CDN/DNS/TLS 本身就有延迟，多一个 9 对体验无意义
4. **100% SLO = Error Budget 为零 → 不敢做任何变更 → 服务反而更脆弱**

| SLO | 月不可用时间 | 适用场景 |
|-----|------------|---------|
| 99.9% | 43 分钟 | 内部服务 |
| 99.95% | 21 分钟 | 对外服务推荐起点 |
| 99.99% | 4.3 分钟 | 需要多 AZ + 自动恢复 |
| 99.999% | 26 秒 | 几乎不可能靠单一团队做到 |

## 二、SLO 的粒度：分层模型

### 2.1 三层 SLO 架构

```
┌──────────────────────────────────────────────────────┐
│  第 1 层：用户旅程级 SLI（2-3 个）                     │
│  "用户从访问到下单成功的端到端可用性 ≥ 99.9%"           │
├──────────────────────────────────────────────────────┤
│  第 2 层：关键服务级 SLO（每个服务独立，5-8 个）         │
│  登录服务 ≥99.95%  │  支付服务 ≥99.99%  │  搜索 ≥99.9%  │
├──────────────────────────────────────────────────────┤
│  第 3 层：基础设施级 SLO（3-5 个）                     │
│  K8s 集群 ≥99.95%  │  数据库 ≥99.99%  │  网络 ≥99.995%  │
└──────────────────────────────────────────────────────┘
```

### 2.2 关键原则

- **不是所有微服务都值得设 SLO**——只给直接承载用户关键路径的服务设
- **基础设施 SLO 必须严于业务 SLO**：支付 99.99% → 数据库必须 ≥ 99.995%
- **依赖链可用性乘法**：你的服务 99.95% × LDAP 99.9% × 网络 99.999% ≈ 用户实际感受到 99.839%

### 2.3 中小团队推荐数量

| 层级 | 数量 | 示例 |
|------|------|------|
| 用户旅程 SLI | 2-3 个 | 下单、登录、核心内容浏览 |
| 关键服务 SLO | 5-8 个 | 登录、支付、订单、搜索、用户中心 |
| 基础设施 SLO | 3-5 个 | K8s、数据库、Redis、消息队列、CDN |
| **合计** | **10-15 个** | |

## 三、Error Budget 核心体系

### 3.1 定义与计算

```
Error Budget = 1 - SLO 目标值

月 Error Budget（分钟）= 30 × 24 × 60 × (1 - SLO)
                        = 43200 × 0.05%
                        = 21.6 分钟（对于 99.95%）
```

**这个预算包括：** 计划内变更引发的中断 + 已知 bug 导致的降级 + 基础设施故障 + 人为操作失误

### 3.2 燃烧速率（Burn Rate）

```
Burn Rate = 实际消耗预算的速度 / 匀速消耗的速度
```

| Burn Rate | 含义 | 如果用光了... |
|-----------|------|--------------|
| 1x | 刚好匀速 | 4 周用完 |
| 2x | 2 倍速度 | 2 周用完 |
| 10x | 10 倍速度 | 2.8 天用完 |
| 100x | 100 倍速度 | ~6.7 小时用完 |

### 3.3 告警矩阵（Google SRE Book 推荐）

| 告警等级 | Burn Rate | 短窗口 | 长窗口 | 含义 |
|---------|-----------|--------|--------|------|
| **Page** 🔥 | ≥ 14.4x | 1 小时 | 6 小时 | 2% 预算 1 小时内耗尽 |
| **Page** 🚨 | ≥ 2x | 6 小时 | 30 分钟 | 预算严重消耗 |
| **Warning** ⚠️ | ≥ 1x | 3 天 | 30 分钟 | 预算消耗偏快 |

> **为什么双窗口？** 短窗口防止漏警，长窗口防止误报。指标偶尔抖动不会触发告警。

### 3.4 决策矩阵

| 剩余预算 | 状态 | 变更策略 |
|---------|------|---------|
| > 70% | 🟢 充裕 | 正常发布，鼓励灰度 + 实验 |
| 50%-70% | 🟡 注意 | 谨慎发布，额外 CR，缩小灰度比例 |
| 20%-50% | 🟠 警告 | 只允许紧急修复 + 安全补丁，冻结功能开发 |
| < 20% | 🔴 危险 | 冻结所有变更，团队全力做稳定性审计 |
| 已耗尽 | ⚫ 违规 | Postmortem，冻结直到预算滚回正值 |

## 四、Prometheus 实战落地

### 4.1 SLI 定义与 Burn Rate 计算

```promql
# SLI：5 分钟窗口可用性
login:sli_availability:ratio_5m = 
  sum(rate(login_requests_total{status="success"}[5m]))
  /
  sum(rate(login_requests_total[5m]))

# Burn Rate 1h
login:error_budget:burn_rate_1h =
  (1 - sum(rate(login_requests_total{status="success"}[1h]))
       / sum(rate(login_requests_total[1h])))
  / 0.0005

# Burn Rate 6h
login:error_budget:burn_rate_6h =
  (1 - sum(rate(login_requests_total{status="success"}[6h]))
       / sum(rate(login_requests_total[6h])))
  / 0.0005
```

### 4.2 三级告警规则

```yaml
# Page: Burn Rate > 14.4x（2% 预算 1 小时内烧完）
- alert: LoginErrorBudgetBurn_Critical
  expr: |
    login:error_budget:burn_rate_1h > 14.4
    and
    login:error_budget:burn_rate_6h > 14.4
  for: 2m
  severity: critical

# Warning: Burn Rate > 2x
- alert: LoginErrorBudgetBurn_Warning
  expr: |
    login:error_budget:burn_rate_1h > 2
    and
    login:error_budget:burn_rate_6h > 2
  for: 5m
  severity: warning

# Info: 月度预算消耗过半
- alert: LoginErrorBudgetHalfSpent
  expr: |
    (1 - sum(rate(login_requests_total{status="success"}[28d]))
         / sum(rate(login_requests_total[28d])))
    > 0.5 * (1 - 0.9995)
  severity: info
```

### 4.3 剩余预算百分比（Grafana 面板）

```promql
1 - (
  (1 - (
    sum(rate(login_requests_total{status="success"}[28d]))
    /
    sum(rate(login_requests_total[28d]))
  ))
  / 0.0005
)
```

### 4.4 CI/CD 门禁

```yaml
deploy_to_prod:
  before_script:
    - |
      BUDGET=$(curl -s prometheus:9090/api/v1/query?query=...)
      if [ $BUDGET -lt 20 ]; then
        echo "❌ Error Budget 剩余 < 20%，禁止部署"
        exit 1
      fi
      if [ $BUDGET -lt 50 ]; then
        echo "⚠️ Error Budget < 50%，需 Team Lead 审批"
      fi
```

## 五、依赖外部服务的 Error Budget 管理

### 5.1 分层记账

在指标上打责任归属标签：

```promql
# 请求按失败原因标记
login_requests_total{status="error", cause="internal"}   # 你的锅
login_requests_total{status="error", cause="ldap"}       # LDAP 的锅
login_requests_total{status="error", cause="sms"}        # SMS 的锅
login_requests_total{status="error", cause="sso"}        # SSO 的锅
```

### 5.2 两个 SLI 并列

| SLI | 公式 | 用途 |
|-----|------|------|
| 👤 用户视角（全链路） | success / all | 对外展示、业务决策 |
| 🏠 自有服务（扣除依赖） | success / (all - 外部依赖失败) | 对内考核、你自己的 SLO |

### 5.3 依赖降级策略

```
登录流程：
  1. 先试 LDAP (99.9%)
  2. LDAP 超时 500ms → 降级用本地缓存（24h 内登录过的用户）
  3. 本地无缓存 → 回退到 SSO
  4. 全部路径失败 → 才算一次真正的失败
```

## 六、多 SLI 统一管理

### 6.1 三种方案

| 方案 | 做法 | 适用阶段 |
|------|------|---------|
| **A: 独立管理** | 每个 SLI 各算各的 Budget，取最差维度做决策 | 起步阶段 |
| **B: 复合 SLI** | 好请求 = 成功 + 延迟达标，合并成一个指标 | 简化阶段 |
| **C: 联合告警** | 独立 Budget + 多维度联合告警规则 | 成熟阶段 |

### 6.2 决策原则：Weakest Link

> 任何一个 SLI 的 Error Budget 进入红灯 → 全体冻结

### 6.3 统一看板

```yaml
Grafana 面板核心：
  title: "现在能发布吗？"
  query: min(各 SLI 剩余预算比例)
  thresholds:
    > 0.7 → 🟢 "可以发，正常流程"
    > 0.5 → 🟡 "可以发，需要审批"
    > 0.2 → 🟠 "除非紧急修复，否则停止"
    > 0.0 → 🔴 "冻结，先修稳定性"
```

## 七、组织落地策略

### 7.1 翻译成老板的语言（💰）

```
SLO 违规损失 = 不可用时间 × 每秒业务价值

每秒业务价值 = (日活 × 转化率 × 客单价) / 86400
```

> "上个月登录挂了 23 分钟，日活 50 万、转化率 30%、客单价 200 块 → **约损失 56 万。** 建议下个月冻结功能开发两周做稳定性。——我把这 56 万省回来。"

### 7.2 嵌入 CI/CD（自动化，不靠自觉）

```
代码 push → CI 通过 → 检查 Error Budget
                        ├── >70% → 自动部署
                        ├── 50-70% → 需要 Lead 审批
                        └── <50% → ❌ 拒绝（紧急修复除外）
```

### 7.3 渐进式推广

1. 选一个"痛过多次但非核心支付"的服务试点
2. 花 2 周搭好 SLI + Budget 看板 + 告警
3. 跑 1 个月，只观察不干预
4. 第 2 个月做 SLO 评审会，拿数据说话
5. 第 3 个月开始按 Budget 做变更决策
6. 3 个月后拿结果给老板看 → 推广

### 7.4 为紧急情况留逃生门

| 场景 | Budget 耗尽时的处理 |
|------|-------------------|
| 安全漏洞 CVE 9.0+ | **直接上线**，事后复盘 |
| 数据丢失风险 | 同上 |
| PM 坚持的功能 | 需 VP/Director 审批 + 回滚方案 |
| 一般 bug fix | 等，或滑到下一窗口 |

### 7.5 团队推广话术

> "我想试行 Error Budget 机制。**这不是给大家加条条框框——有了它，你想发版就有明确的权利。** 只要预算充足，你可以自己判断能不能上。我们先用搜索服务试 3 个月，不影响其他服务。我承诺：**第一，不拿告警来事后责怪任何人。第二，如果有让团队变慢的地方，我们一起改。**"

## 八、常见陷阱

| 陷阱 | 症状 | 纠正 |
|------|------|------|
| 只看 Burn Rate 不看绝对值 | 1x 持续 3 周没人管 | 同时监控 Burn Rate 和剩余预算量 |
| 低流量误报 | 夜间一个失败就把 Burn Rate 打到 100x | 加最小流量阈值 |
| 人为关闭告警 | "这周上线多，先关了" | 绝对禁止。如果经常破说明 SLO 定太高 |
| 月底故意花预算 | 预算剩很多，故意停服消耗 | 多余预算说明可以提速或提 SLO |
| 全局 SLO 掩盖局部灾难 | 首页正常但支付挂了，全局没破 | 关键服务必须独立设 SLO |
| SLO 太多太碎 | 50 个服务各一个，没人看 | 10-15 个足矣，先做关键的 |

## 九、SLO 定义模板

```yaml
service: login-service
slos:
  - sli: 可用性
    description: "成功响应(非5xx) / 所有 login API 请求"
    metric: login_requests_total{status!~"5.."} / login_requests_total
    target: 99.95%
    compliance_windows:
      - duration: 28d
        target: 99.95%
      - duration: 1h
        target: 99.5%          # 防止小时级长时间中断
    measurement_lag: 30m        # 允许指标延迟 30 分钟
    owners:
      - team: auth-team
        oncall: "https://pager.example.com/auth"
```
